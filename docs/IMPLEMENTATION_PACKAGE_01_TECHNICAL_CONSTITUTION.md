# Purpose

The first one hundred builds defined **what** Citizens United Facts will become.

The next fifty implementation packages define **exactly how it will be built.**

This document becomes the technical constitution for every developer, AI system, Cursor session, and future contributor.

Every line of code should align with this document.

---

# The Prime Directive

Build **one integrated institution**, not a collection of disconnected applications.

Everything must feel like one platform.

- One login.
- One design language.
- One data model.
- One knowledge platform.
- One AI ecosystem.
- One Mission Control.
- One Arkansas.

---

# Development Philosophy

Every feature should answer three questions:

## Why does this exist?

Mission.

## Who does it serve?

User.

## How does it strengthen the institution?

System.

If a feature cannot answer all three questions, it should not be built.

---

# Platform Principles

The platform shall be:

Volunteer-first · Mobile-first · Accessibility-first · Research-first · Evidence-first · AI-assisted · Mission-Control-driven · Arkansas-focused · Future-ready.

---

# Technical Principles

Every system should be:

Modular · Reusable · Documented · Searchable · Observable · Secure · Scalable · Maintainable · Testable.

No shortcuts that create future technical debt.

---

# User Principles

Every user should immediately understand:

- Where they are.
- What they can do.
- What they should do next.

No page should ever leave a user wondering what comes next.

---

# Mission Control First

Every major feature must expose operational information to Mission Control.

Mission Control is not an afterthought. Mission Control is part of every feature.

Examples include:

Learning progress · Volunteer activity · County growth · Research completion · Coalition health · Technology health · Institution readiness.

If Mission Control cannot observe it, it probably has not been designed completely.

---

# AI First

Every feature should ask: *How can AI make this easier?*

Examples: Research · Writing · Searching · Planning · Teaching · Summarizing · Scheduling · Volunteer support.

Never replace human judgment. Always increase human capability.

---

# Knowledge First

The institution owns knowledge. Every feature either:

Creates knowledge · Organizes knowledge · Uses knowledge · Protects knowledge · Or teaches knowledge.

Knowledge is the institution's most valuable asset.

---

# Arkansas First

Whenever possible, begin with Arkansas.

County → City → Neighborhood → Community.

The institution is designed specifically for Arkansas before considering broader expansion.

---

# Volunteer First

Every workflow should minimize volunteer effort.

Questions to ask:

- Can AI help?
- Can automation help?
- Can documentation help?
- Can Mission Control reduce manual work?

Every saved volunteer hour strengthens the institution.

---

# Simplicity Rule

Simple beats clever. Readable beats complex. Consistent beats unique.

Future volunteers should understand the system quickly.

---

# The Four-Layer Architecture

Every feature belongs to one layer.

| Layer | Name | Scope |
|-------|------|-------|
| **One** | Experience | User Interface |
| **Two** | Application | Business Logic |
| **Three** | Knowledge | Data & Documents |
| **Four** | Institution | Mission Control & Governance |

Every feature should clearly identify its primary layer.

---

# Documentation Rule

Every major module must include:

Purpose · Owner · Dependencies · Data used · Mission Control integration · AI integration · Related systems.

Nothing important exists without documentation.

---

# Naming Standards

Use human-readable names.

Examples: County Dashboard · Volunteer Profile · Research Library · Evidence Ledger · Community Conversation · Education Leader · Mission Control.

Avoid unnecessary technical jargon in user-facing features.

---

# Build Order Rule

Every future implementation package should follow this sequence:

Design → Data → API → Business Logic → User Interface → Mission Control → AI → Testing → Documentation → Acceptance.

Nothing skips the sequence.

---

# Success Definition

Implementation Package #1 is complete when every future build has a single governing doctrine that defines:

- How the institution is built.
- How developers think.
- How AI contributes.
- How volunteers are served.
- How Mission Control operates.
- How Arkansas remains the focus.
